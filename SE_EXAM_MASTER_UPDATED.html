Software Engineering

8-10 Marks

Software Engineering is the systematic, disciplined, and quantifiable approach to the development, operation, and maintenance of software. It applies engineering principles to software creation to produce correct, reliable, and maintainable software within budget and on schedule.

4 Layers of Software Engineering

🏆 Quality Focus (Foundation)
↑
📋 Process (Framework)
↑
⚙️ Methods (Technical How-To)
↑
🔧 Tools (Automation Support)

1. Quality Focus (Foundation)

The most critical layer. Every SE activity aims for quality. Quality ensures reliability, maintainability, and user satisfaction.

2. Process

The framework that defines how software is developed (requirements → design → build → test → deploy → maintain). Examples: Waterfall, Spiral, Agile.

3. Methods

Technical procedures for each phase. Examples: Requirements analysis techniques, design patterns, testing strategies, coding standards.

4. Tools

Automated support for process and methods. Examples: IDEs, CASE tools, version control, testing frameworks, CI/CD pipelines.

Qualities of Good Software (FURPS)

  • Functionality: Does it do what users need?
  • Usability: Is it easy to use?
  • Reliability: Does it work consistently without crashes?
  • Performance: Is it fast and efficient?
  • Supportability: Can it be maintained and updated?
✅ Advantages of SE
  • Improves software quality
  • Reduces development cost
  • Ensures on-time delivery
  • Easier maintenance
  • Better documentation
❌ Challenges
  • High initial cost
  • Requires trained team
  • Complex process
  • Time-consuming
Always draw the 4-layer pyramid diagram. Every question about SE basics can be answered using these layers. Examiners reward clear diagrams.

Software Myths

5 Marks

Software myths are false beliefs held by managers, customers, and practitioners that lead to poor decisions and project failures.

Management Myths

  • Myth: Having a book of standards ensures quality
    Reality: Standards alone don't ensure quality. The team must follow and understand them.
  • Myth: Adding more programmers speeds up a late project
    Reality: Adding people mid-project slows it down (Brooks' Law)
  • Myth: We can outsource and relax
    Reality: Outsourcing still requires active management and oversight

Customer Myths

  • Myth: General objectives are enough; details come later
    Reality: Ambiguous requirements cause massive rework. Details are essential upfront.
  • Myth: Software is flexible; change anytime
    Reality: Late changes are very costly. Early requirement stability is critical.

Practitioner Myths

  • Myth: Once code works, the job is done
    Reality: Maintenance takes 60–80% of total software cost over its lifetime.
  • Myth: Documentation is not important
    Reality: Documentation is crucial for maintenance and team handoffs.
Questions ask: "Explain any 3 myths and their reality." Pick from the three categories and explain clearly.

SDLC – Software Development Life Cycle

10 Marks ⭐

SDLC is a structured process that defines the phases involved in software development from initial concept to final delivery and maintenance. It provides a framework for planning, creating, testing, and delivering software systems.

6 SDLC Phases

1️⃣ Requirement Analysis
→
2️⃣ Design
→
3️⃣ Implementation
→
4️⃣ Testing
→
5️⃣ Deployment
→
6️⃣ Maintenance

Phase 1: Requirement Analysis

Gather what users and stakeholders need from the system. Output: Software Requirements Specification (SRS) document.

Phase 2: System Design

Create the architecture and detailed design (database schema, UI mockups, API specifications). Output: Design Document.

Phase 3: Implementation (Coding)

Write the actual code in chosen programming language(s) based on the design.

Phase 4: Testing

Test for bugs and ensure quality. Unit → Integration → System → Acceptance testing.

Phase 5: Deployment

Release to end users (production environment).

Phase 6: Maintenance

Fix bugs, apply patches, add enhancements. This phase lasts the longest.

Draw the SDLC waterfall diagram with all 6 phases. Label each phase clearly. Examiners award marks even if explanation is brief.

Waterfall Model

8 Marks

Waterfall is the first and most classic SDLC model. It is sequential and linear — each phase must be completed before the next begins. No going back, like water flowing downward.

Waterfall Flow

Requirements
↓
Design
↓
Implementation
↓
Testing
↓
Deployment
↓
Maintenance
✅ Advantages
  • Simple and easy to understand
  • Well-defined phases & milestones
  • Easy to manage small projects
  • Good documentation
❌ Disadvantages
  • No customer feedback mid-process
  • Testing only at the end
  • Cannot accommodate changes
  • High risk for complex projects

When to Use Waterfall

  • Requirements are clear and fixed upfront
  • Project scope is well-defined
  • Small to medium projects
  • Technology is stable and known
  • Team is not distributed

Extreme Programming (XP)

5 Marks

Extreme Programming is an Agile software development methodology that emphasizes technical excellence and good software engineering practices through short development cycles and continuous customer feedback.

Key XP Practices

  • Test-Driven Development (TDD): Write tests BEFORE writing code. Red-Green-Refactor cycle.
  • Pair Programming: Two developers work together on one machine, improving code quality and knowledge sharing
  • Continuous Integration: Code is integrated and tested multiple times per day
  • Refactoring: Continuously improve code structure without changing functionality
  • Small Releases: Frequent, small releases with working features
  • Simple Design: Do the simplest thing that works; avoid over-engineering
  • Collective Ownership: Any team member can modify any code (with tests)
  • Coding Standards: Follow agreed conventions for consistency
XP is often asked in comparison with other methodologies. Know TDD (Test-Driven Development) as the core practice. Real-world XP is used at Google, Facebook, and many startups.

Project Planning & Scheduling

5 Marks

Software project planning involves estimating resources, timelines, costs, and tasks required to complete a project successfully.

Key Planning Activities

  • Scope Definition: Define what will and won't be built
  • Work Breakdown Structure (WBS): Break project into smaller, manageable tasks
  • Effort Estimation: Estimate person-months or person-hours for each task
  • Scheduling: Assign tasks to timeline (Gantt charts, PERT/CPM)
  • Resource Allocation: Assign team members to tasks
  • Risk Planning: Identify and plan for potential risks
  • Cost Estimation: Calculate project budget

Estimation Metrics

  • LOC (Lines of Code): Size metric — total non-comment, executable lines
  • KLOC: 1000 lines of code
  • Function Points (FP): Measures functionality delivered, language-independent
  • COCOMO Model: Effort = a × (KLOC)^b (for effort estimation)
Questions ask about WBS (Work Breakdown Structure) and estimation techniques. Understand the difference between effort estimation and schedule estimation.

Requirements Engineering

10 Marks ⭐

Requirements Engineering is the process of defining, documenting, and maintaining software requirements. It involves understanding stakeholder needs and documenting them in a Software Requirements Specification (SRS) document.

RE Process (6 Steps)

1️⃣ Inception
→
2️⃣ Elicitation
→
3️⃣ Elaboration
→
4️⃣ Negotiation
→
5️⃣ Specification
→
6️⃣ Validation

Inception:

Understand the basic problem and identify stakeholders.

Elicitation:

Gather requirements via interviews, surveys, observations, brainstorming sessions.

Elaboration:

Create analysis models (use cases, data flow diagrams, ERD). Understand system interactions.

Negotiation:

Resolve conflicting requirements between stakeholders. Prioritize features.

Specification:

Write formal SRS (Software Requirements Specification) document with all requirements in detail.

Validation:

Review SRS with stakeholders to ensure it's correct, complete, consistent, and testable.

SRS Contents

  • System overview and objectives
  • Functional requirements (features)
  • Non-functional requirements (performance, security, etc.)
  • Use cases and scenarios
  • Data models and interfaces
  • Constraints and assumptions

Functional vs Non-Functional Requirements

5 Marks
Aspect Functional Requirements (FR) Non-Functional Requirements (NFR)
Focus What system does How system performs
Examples Login, Search, Make payment, Generate reports Response time < 2 sec, 99.9% uptime, Support 1000 users
Testing Easy to test (pass/fail) Difficult to measure precisely
Source User/business needs System constraints & quality attributes
Change impact High impact on design Affects architecture decisions

FURPS+ (NFR Classification)

  • F - Functionality: What the system does (actually FR)
  • U - Usability: Ease of use, UI quality, accessibility
  • R - Reliability: Availability, failure frequency, recovery
  • P - Performance: Response time, throughput, scalability
  • S - Supportability: Maintainability, portability, compatibility
  • + others: Security, compliance, legal constraints
FURPS is frequently asked. Remember: FR tells WHAT, NFR tells HOW GOOD. Always give examples when differentiating.

Feasibility Study

5 Marks

A feasibility study evaluates whether a proposed project is technically possible, economically viable, operationally practical, and legally/ethically acceptable before committing significant resources.

4 Types of Feasibility

1. Technical Feasibility

Can we build it with available technology? Do we have the required tools, platforms, expertise, and infrastructure?

  • Is technology mature and stable?
  • Do we have skilled developers?
  • Are third-party libraries available?

2. Economic Feasibility

Is the project worth the cost? Calculate ROI (Return on Investment).

  • Project cost vs. expected benefits
  • Payback period
  • Cost-benefit analysis

3. Operational Feasibility

Will users accept and use the system? Is it practical for day-to-day operations?

  • User training requirements
  • Change management needs
  • Organizational readiness

4. Schedule Feasibility

Can we deliver the project on time? Do we have realistic deadlines?

  • Project duration estimate
  • Resource availability
  • Dependency analysis
Feasibility study is conducted BEFORE development. If any feasibility is questionable, the project may be rejected or postponed.

UI Design & Golden Rules

5 Marks

User Interface (UI) Design is the process of creating interfaces that allow users to interact with software effectively and pleasantly. Good UI follows established principles to ensure usability.

Shneiderman's 8 Golden Rules of UI Design

  1. Strive for consistency: Use same operations, terminology, colors for similar tasks
  2. Enable frequent users to use shortcuts: Keyboard shortcuts, command abbreviations
  3. Offer informative feedback: System should communicate what it's doing
  4. Design dialog to yield closure: Clear sequence with beginning, middle, end
  5. Prevent errors: Design to prevent problems before they occur
  6. Permit easy reversal of actions: Provide undo/redo functionality
  7. Support internal locus of control: Users should feel in control, not controlled by system
  8. Reduce short-term memory load: Don't require users to remember info; show it

Architectural Styles

  • MVC (Model-View-Controller): Separates data, presentation, and logic
  • Layered Architecture: Presentation → Business Logic → Data layers
  • Repository Pattern: Central data store accessed by components
  • Client-Server: Server provides services to multiple clients
Questions ask to explain Shneiderman's rules or Golden Rules. Just list all 8 with brief explanation. Also be ready to explain good UI vs bad UI examples.

Unit & Integration Testing

5 Marks

Unit testing tests individual components in isolation. Integration testing tests how components interact.

Unit Testing

  • Tests individual functions, methods, or modules
  • Done in isolation from rest of system
  • Uses test cases with expected inputs and outputs
  • Done by developers
  • Fast and catches bugs early

Test Drivers & Stubs

  • Test Driver: A dummy program that calls the module being tested and supplies test data
  • Stub: A dummy program that simulates the behavior of a called module (returns dummy results)

Integration Testing Approaches

Approach Method Pros Cons
Top-Down Start from main, go down. Uses stubs. Tests control flow early Need many stubs
Bottom-Up Start from leaf modules, go up. Uses drivers. Real modules tested thoroughly Delays control flow testing
Big Bang Integrate all at once and test Simple Hard to find bugs
Questions ask to compare Top-Down vs Bottom-Up. Know the differences in tools used (stubs vs drivers) and what's tested first in each approach.

Black Box vs White Box Testing

10 Marks ⭐
Aspect Black Box White Box
Also called Functional/Behavioral testing Structural/Glass Box testing
Code visibility No access to source code Full access to source code
Focus Input/Output Code structure & logic
Tester background QA testers (non-technical okay) Developers
Techniques Equivalence partitioning, Boundary value analysis Basis path, Loop testing, Statement/branch coverage
Dependency Independent of implementation Depends on code structure

Black Box Techniques

  • Equivalence Partitioning: Divide inputs into valid/invalid classes, test one from each
  • Boundary Value Analysis: Test values at edges (minimum, maximum, just below/above)
  • Error Guessing: Based on experience, guess likely error cases

White Box Techniques

  • Statement Coverage: Ensure every statement is executed at least once
  • Branch Coverage: Ensure every if/else branch is taken
  • Basis Path Testing: Use flowchart to find all independent paths
  • Loop Testing: Test loops (zero iterations, one iteration, multiple)

Cyclomatic Complexity

Measures code complexity — tells how many independent paths exist through code.

Formula: V(G) = E - N + 2P
E = edges, N = nodes, P = connected components (usually 1)
  • V(G) = 1: Simple, single path
  • V(G) ≤ 10: Acceptable complexity
  • V(G) > 10: High complexity, high-risk code, needs lots of testing
Cyclomatic Complexity formula is frequently asked. Know how to calculate V(G) from a given flowchart. Also be ready to compare Black Box and White Box with real examples.

Art of Debugging

5 Marks

Debugging is the process of finding and fixing errors (bugs) in a program. It begins when a test case uncovers an error.

Debugging Strategies

1. Brute Force (Print Statements)

Add print statements or use debugger to output variable values throughout execution. Most common but least efficient.

  • ✅ Simple, easy to implement
  • ❌ Time-consuming, error-prone

2. Backtracking

Start from where the error occurs and trace backwards through code to find the root cause.

  • ✅ Effective for simple programs
  • ❌ Difficult for large programs

3. Cause Elimination

List all possible causes, design test cases to eliminate each one systematically (like binary search).

  • ✅ Logical, systematic approach
  • ✅ Works well for complex programs
Questions ask to compare and explain these 3 debugging strategies. Provide examples for each approach.

Software Metrics

10 Marks ⭐

Software metrics are quantitative measures used to assess software quality, productivity, complexity, and progress. They help track projects and identify problem areas.

Key Metrics

Size Metrics

  • LOC (Lines of Code): Simple size metric — non-comment, executable lines
  • KLOC: 1000 lines of code
  • Function Points (FP): Measures functionality, language-independent

Complexity Metrics

  • Cyclomatic Complexity V(G) = E - N + 2P: Number of independent paths
  • Halstead Complexity: Based on number of operators and operands

Quality Metrics

  • Defect Density = Defects / KLOC: Defects per 1000 lines (lower is better)
  • MTBF = MTTF + MTTR: Mean Time Between Failures = Uptime + Downtime
  • Reliability R(t) = e^(-λt): Exponential reliability model
  • Code Coverage: % of code executed during testing

Productivity Metrics

  • Productivity = LOC Delivered / Person-Month: How much code per developer per month
Metric Type What It Measures Examples
Analysis Metrics Size of requirements Function Points, Use Case Points
Design Metrics Design quality Coupling, Cohesion, Depth of Inheritance
Code Metrics Code complexity LOC, Cyclomatic Complexity
Testing Metrics Test effectiveness Defect Density, Code Coverage
Maintenance Metrics Maintenance effort MTBF, Change Request Rate
Know Cyclomatic Complexity formula and Defect Density formula. Questions ask to calculate these from given data. Always show step-by-step calculations.

Software Risk Management

10 Marks ⭐

Risk is any condition or event that may cause harm to a software project in terms of schedule, budget, quality, or performance. Risk Management involves identifying, assessing, and planning responses to risks.

Types of Risks

  • Schedule Risk: Project delays, missing deadlines
  • Budget Risk: Cost overruns, insufficient funding
  • Technical Risk: Technology limitations, complex architecture
  • Operational Risk: Hardware failures, user errors, environment issues
  • Programmatic Risk: External constraints, contracts, organizational changes

Reactive vs Proactive Risk Management

Aspect Reactive (After Risk Occurs) Proactive (Before Risk Occurs)
Approach "Fire fighting" mode Prevention-first approach
Cost Higher Lower
Planning No risk plan Formal RMMM plan
When After problem occurs During planning phase

Risk Exposure Calculation

Formula: RE = P(UO) × C(UO)
P(UO) = Probability of unsatisfactory outcome
C(UO) = Cost/Impact if risk occurs
Proactive risk management is always better than reactive. Know the RMMM plan (next topic). Always contrast the two approaches in 8-10 mark questions.

RMMM – Risk Mitigation, Monitoring & Management

10 Marks ⭐

RMMM is a formal plan for handling software risks. It defines how each identified risk will be mitigated, monitored, and managed throughout the project.

RMMM Process Flow

1️⃣ Risk Identification
→
2️⃣ Risk Projection
→
3️⃣ Risk Mitigation
→
4️⃣ Risk Monitoring
→
5️⃣ Risk Management

1. Risk Identification

Discover all potential risks through brainstorming, expert interviews, historical data, checklists.

2. Risk Projection (Assessment)

Estimate probability and impact of each risk. Calculate risk exposure (RE = P × C).

3. Risk Mitigation

Define strategies to reduce probability or impact of each risk. Preventive actions.

  • Avoid risk (change scope/technology)
  • Reduce probability (training, better tools)
  • Reduce impact (contingency plans, redundancy)

4. Risk Monitoring

Track indicators showing if a risk is becoming real. Watch for "risk triggers."

  • Regular risk review meetings
  • Track key indicators
  • Adjust mitigation strategies if needed

5. Risk Management (Contingency Planning)

Execute contingency plans when risks actually occur. Have backup strategies ready.

Risk Table: A document that lists all identified risks with their probability, impact, RE, mitigation strategy, and monitoring triggers.
Questions ask to explain RMMM or create a risk table for a project scenario. Know all 5 steps and be able to explain each with examples.

Software Quality & Formal Technical Review (FTR)

10 Marks ⭐

Software quality is the degree to which software meets specified requirements and satisfies user needs. Formal Technical Review (FTR) is a structured method to ensure quality through peer review.

Quality Dimensions (FURPS+)

  • Functionality: Does it do what users need?
  • Usability: Is it easy to use?
  • Reliability: Does it work consistently?
  • Performance: Is it fast and efficient?
  • Supportability: Can it be maintained?
  • + Security, Compliance, Portability

Formal Technical Review (FTR)

An FTR is a structured meeting where technical experts review software work products to find errors and ensure compliance with standards.

FTR Goals

  • Uncover errors in function, logic, or implementation
  • Verify compliance with requirements and standards
  • Ensure code follows best practices
  • Make development more uniform and manageable

FTR Meeting Guidelines

  • Review the product, not the producer
  • Set an agenda and maintain focus
  • Limit debate and rebuttal
  • Maximum 2-hour meeting duration
  • Focus on finding problems, not solving them
  • Document all issues found

Types of Reviews

Type Formality Participation Focus
Walkthrough Informal Author leads, peers attend Understanding & general issues
Inspection Formal (FTR) Trained inspectors, author present Defect finding, metrics
Technical Review Formal Technical experts, management Compliance, correctness, standards
FTR is a preventive quality technique (finds errors BEFORE testing). Questions ask to explain FTR goals, guidelines, or compare with other review types.

Statistical Software Quality Assurance (SSQA)

5 Marks

SSQA involves collecting and analyzing defect data to identify the causes of software errors and improve the development process systematically.

SSQA Process

  1. Collect defect information from all sources (testing, production, users)
  2. Classify each defect by cause (incomplete spec, misunderstood requirement, implementation error, etc.)
  3. Analyze using Pareto Analysis
  4. Focus improvement efforts on vital few causes
  5. Track and verify improvements

Pareto Analysis (80/20 Rule)

80% of software defects come from 20% of causes. Focus resources on fixing those vital few causes for maximum impact.

  • Create Pareto chart: defect categories vs frequency
  • Identify the "vital few" categories
  • Allocate resources to top causes
  • Re-measure after improvements
SSQA and Pareto Analysis are often asked together. Questions ask: "How is Pareto Analysis used in SSQA to improve quality?" Answer with the 80/20 concept and example.

CASE Tools – Computer-Aided Software Engineering

10 Marks ⭐

CASE tools are software applications that help developers in designing, developing, testing, and maintaining software. They automate and support SE activities to improve productivity and quality.

Classification of CASE Tools

Upper CASE Tools (Early Phase Support)

Support planning and analysis phases. Help with requirements gathering and system design.

  • Requirement collection tools
  • Data modeling (ER diagrams)
  • UML diagram tools (StarUML, Visio)
  • System design documentation
  • DFD (Data Flow Diagram) creation

Lower CASE Tools (Late Phase Support)

Support implementation, testing, and maintenance phases.

  • IDEs (Eclipse, VS Code) — Code writing
  • Compilers and debuggers
  • Testing tools (Selenium, JUnit)
  • Version control (Git, GitHub)
  • Build tools (Maven, Gradle)

Integrated CASE Tools (Full SDLC)

Combine Upper and Lower CASE. Cover entire software lifecycle in one tool.

  • Rational Rose (IBM) — Modeling and development
  • Enterprise Architect (Sparx)
  • Microsoft Visual Studio
  • JetBrains IntelliJ IDEA

UML Diagram Overview

10 Marks ⭐

UML (Unified Modeling Language) is a standard visual modeling language used to represent the structure and behavior of a software system. It helps developers, analysts, and users understand the system before implementation.

Why UML is used

  • Visualizes system requirements and design
  • Improves communication among stakeholders
  • Helps in documentation and maintenance
  • Supports object-oriented analysis and design

Structural Diagrams

These diagrams describe the static structure of the system: classes, components, nodes, objects, and packages.

Class Diagram

Shows classes, attributes, methods, and relationships.

classDiagram
class Student{
  +name
  +rollNo
  +enroll()
}
class Course{
  +code
  +title
}
Student --> Course : enrolls
GFG link

Object Diagram

Shows a snapshot of objects at a specific moment.

flowchart LR
  obj1["student1:Student"]
  obj2["course1:Course"]
  obj1 -->|registered in| obj2
GFG link

Component Diagram

Shows system components and their dependencies.

flowchart LR
  UI[UI Component] --> API[API Component]
  API --> DB[(Database)]
GFG link

Deployment Diagram

Shows how software is deployed onto hardware nodes.

flowchart LR
  Client[Client Node] --> Web[Web Server]
  Web --> App[App Server]
  App --> DB[(DB Server)]
GFG link

Package Diagram

Groups related classes into packages and shows dependencies.

flowchart TB
  subgraph UI[UI Package]
    A[Login]
    B[Dashboard]
  end
  subgraph Core[Core Package]
    C[Business Logic]
    D[Data Access]
  end
  UI --> Core

Behavioral Diagrams

These diagrams describe system behavior, interaction, and workflow over time.

Use Case Diagram

Shows actors interacting with system functions.

flowchart LR
  User([User]) --> Login((Login))
  User --> Search((Search))
  User --> Buy((Purchase))
GFG link

Sequence Diagram

Shows message flow between objects in time order.

sequenceDiagram
  actor User
  participant UI
  participant Auth
  User->>UI: login()
  UI->>Auth: validate()
  Auth-->>UI: success
  UI-->>User: dashboard
GFG link

Activity Diagram

Represents a workflow or process flow.

flowchart TD
  Start([Start]) --> Check{Valid?}
  Check -- Yes --> Process[Process Data]
  Process --> End([End])
  Check -- No --> Reject[Show Error]
  Reject --> End
GFG link

State Diagram

Shows transitions between states of an object.

stateDiagram-v2
  [*] --> Idle
  Idle --> Active: start
  Active --> Idle: stop
GFG link

Communication Diagram

Shows interaction flow between objects with numbered messages.

flowchart LR
  A[User] -->|1: request| B[UI]
  B -->|2: validate| C[Controller]
  C -->|3: store| D[(DB)]
GFG link
Quick revision: Structural diagrams explain what the system looks like; behavioral diagrams explain how the system behaves.
